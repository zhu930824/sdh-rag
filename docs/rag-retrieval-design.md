# RAG 检索召回设计文档

## 概述

本文档详细描述 SDH-RAG 系统的检索召回流程，包括查询处理、多路召回、混合检索、重排序等核心环节。

## 整体流程

```
用户查询
    ↓
1. 查询改写（多轮对话场景）
    ↓
2. 查询扩展（生成语义变体）
    ↓
3. HyDE 假设性文档生成
    ↓
4. 多查询混合检索（向量 + 关键词）← 相似度阈值在此阶段生效
    ↓
5. RRF 多路结果合并
    ↓
6. Rerank 重排序
    ↓
返回结果
```

---

## 阶段详解

### 1. 查询改写（Query Rewrite）

**触发条件**：开启多轮对话且存在历史记录

**目的**：将依赖上下文的省略问句改写为完整独立查询

**示例**：
```
历史: "什么是微服务？"
用户: "它有什么优点？"
改写: "微服务架构有什么优点？"
```

**实现**：调用 LLM 根据对话历史补全上下文信息

---

### 2. 查询扩展（Query Expansion）

**触发条件**：知识库配置 `enableQueryExpansion = true`

**目的**：生成语义相关的查询变体，提升召回覆盖率

**示例**：
```
原始查询: "如何优化数据库查询性能"
扩展查询:
  - 数据库查询速度慢怎么解决
  - SQL 性能调优方法
  - 数据库索引优化技巧
```

**参数**：
- `queryExpansionCount`：扩展数量，默认 3

---

### 3. HyDE（Hypothetical Document Embedding）

**触发条件**：知识库配置 `enableHyde = true`

**目的**：生成假设性答案文档，用其向量检索，解决"答案词汇与问题词汇不匹配"问题

**原理**：
```
问题 → LLM 生成假设答案 → 答案向量化 → 检索相似文档
```

**优势**：问题与答案的语言风格差异大时效果显著

---

### 4. 混合检索（Hybrid Search）

每个查询（原始 + 扩展 + HyDE）都执行双路检索：

#### 4.1 向量检索（语义召回）

- **原理**：Embedding 向量余弦相似度
- **优势**：捕捉语义相似性，支持同义词、近义表达
- **参数**：`vectorTopK`（默认 10）、`vectorWeight`（默认 0.7）

#### 4.2 关键词检索（精确召回）

- **原理**：BM25 全文检索
- **优势**：精确匹配关键词，对专有名词效果更好
- **参数**：`keywordTopK`（默认 10）、`keywordWeight`（默认 0.3）

#### 4.3 相似度阈值过滤

**在向量检索阶段生效**：当配置了 `similarityThreshold > 0` 时，向量相似度分数低于阈值的结果会被直接过滤，不会进入后续流程。

**阈值说明**：
- Elasticsearch KNN 搜索返回的 score 基于余弦相似度
- 有效范围通常为 0 到 1（值越高越相似）
- 建议阈值：0.3~0.7

| 阈值 | 效果 | 适用场景 |
|-----|------|---------|
| 0.3 | 宽松 | 召回优先，结果较多 |
| 0.5 | 平衡 | 通用场景 |
| 0.7 | 严格 | 精度优先，结果较少 |

#### 4.4 双路召回互补性

| 场景 | 向量检索 | 关键词检索 |
|------|---------|-----------|
| "如何部署应用" | ✅ 语义匹配 | ❌ 可能遗漏 |
| "Kubernetes Pod" | ❌ 可能召回无关内容 | ✅ 精确匹配 |
| "系统配置文件路径" | ✅ 理解意图 | ✅ 精确路径 |

---

### 5. RRF 多查询结果合并

**问题**：多个查询会召回重复文档，需要合并去重并排序

**算法**：Reciprocal Rank Fusion（倒数排名融合）

```
RRF(d) = Σ queryWeight / (k + rank(d))
```

**参数设计**：

| 参数 | 值 | 说明 |
|-----|-----|------|
| k（平滑常数） | 10 | 分数区间约 0.18~0.09，区分度明显 |
| 原始查询权重 | 2.0 | 主要召回来源 |
| 扩展查询权重 | 1.0 | 辅助召回来源 |

**分数示例**（k=10）：

| 来源 | 排名 | 分数 |
|-----|------|------|
| 原始查询 | 1 | 2.0/11 = 0.18 |
| 原始查询 | 5 | 2.0/15 = 0.13 |
| 扩展查询 | 1 | 1.0/11 = 0.09 |
| 扩展查询 | 5 | 1.0/15 = 0.07 |

**核心价值**：
- 被多个查询同时召回的文档得分累加，排名靠前
- 体现"共识投票"思想，多路召回一致认为相关的文档更可信
- **仅用于排序，不用于过滤**（分数量级不适合设阈值）

---

### 6. Rerank 重排序

**触发条件**：知识库配置了 `rankModel`

**目的**：用专门的排序模型对初步结果精细化重排

**流程**：
```
混合检索结果（Top K）
    ↓
Rerank 模型计算 query-document 相关分
    ↓
按 rerank_score 重新排序
```

**分数特点**：
- 输出范围 0-1
- 语义相关性评分

**推荐模型**：
- 阿里 DashScope：`gte-rerank`、`qwen3-rerank`

## 配置参数汇总

| 参数 | 默认值 | 说明 |
|-----|-------|------|
| `enableRewrite` | false | 多轮对话改写开关 |
| `enableQueryExpansion` | false | 查询扩展开关 |
| `queryExpansionCount` | 3 | 扩展查询数量 |
| `enableHyde` | false | HyDE 开关 |
| `vectorTopK` | 10 | 向量检索返回数 |
| `keywordTopK` | 10 | 关键词检索返回数 |
| `vectorWeight` | 0.7 | 向量检索权重 |
| `keywordWeight` | 0.3 | 关键词检索权重 |
| `rankModel` | - | 重排序模型 ID |
| `similarityThreshold` | 0（不过滤） | 向量相似度阈值 |

---

## 关键设计决策

### Q1: 相似度阈值应在哪个阶段生效？

**在向量检索阶段生效**，而不是在 rerank 之后。原因：

1. **效率更高**：低分文档在召回阶段就被过滤，减少后续处理开销
2. **语义更准确**：向量相似度直接反映查询与文档的语义相关性
3. **rerank 不是必须的**：有些场景不开启 rerank，需要在检索阶段就能过滤

### Q2: 为什么 RRF 分数不过滤？

RRF 分数是排名的函数，取值范围取决于 k 值和权重设计（当前约 0.07~0.18），不是概率或相似度，不具备绝对意义。它只用于排序，不适合设阈值。

### Q3: k=10 是怎么选的？

k 值影响分数区间宽度：
- k=60：分数区间 0.016~0.014，几乎无区分度
- k=10：分数区间 0.18~0.09，区分度明显

k 取 `topK` 同量级（如 topK=10，k=10）能保证排名靠前的文档显著高于靠后文档。

### Q4: 为什么原始查询权重 2.0？

用户输入的原始查询是最可信的召回来源，扩展查询和 HyDE 文档作为补充。2:1 的权重比让原始查询召回的文档排名优势明显，但扩展查询若多次召回同一文档仍可超越。

## 流程示意

```
Query: "如何优化数据库性能"
    ↓
[启用了 Query Expansion + HyDE]
    ↓
Queries:
  - Q1: "如何优化数据库性能" (原始, weight=2.0)
  - Q2: "数据库查询慢怎么解决" (扩展, weight=1.0)
  - Q3: "SQL调优方法" (扩展, weight=1.0)
  - Q4: <假设性文档内容...> (HyDE, weight=1.0)
    ↓
每个 Query 执行 hybridSearch(vector + keyword, minScore=0.5):
  
  向量检索时过滤：
    docA: score=0.85 ✓
    docB: score=0.72 ✓
    docC: score=0.45 ✗ (被过滤)
    docD: score=0.38 ✗ (被过滤)
    docE: score=0.65 ✓
  
  Q1 → [docA(1), docB(3), docE(5)]
  Q2 → [docA(2), docE(4)]
  Q3 → [docB(2), docE(1)]
  Q4 → [docE(3)]
    ↓
RRF 合并（去重 + 累加分数）:
  docA: 2.0/11 + 1.0/12 = 0.18 + 0.08 = 0.26
  docB: 2.0/13 + 1.0/12 = 0.15 + 0.08 = 0.23
  docE: 2.0/15 + 1.0/14 + 1.0/11 + 1.0/13 = 0.39  ← 多路召回最高
    ↓
排序结果: [docE, docA, docB]
    ↓
Rerank (rankModel):
  docE: 0.88
  docA: 0.82
  docB: 0.75
    ↓
返回 Top 3
```
